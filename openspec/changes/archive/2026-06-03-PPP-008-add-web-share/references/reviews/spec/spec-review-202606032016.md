# Review: PPP-008-add-web-share

- **Reviewer**: openspec-reviewer

- **Iteration**: 1 / 10
- **Result**: FAIL

## Summary

Web Share API による共有機能の proposal。フォーマットは健全で `openspec validate --strict` もパスする。ただし、対象コードでは画像が配列 (`images: ImageData[]`) であるにもかかわらず spec/proposal が単数前提で記述されており複数画像の扱いが未定義、共有ボタンの活性条件（テキスト・画像とも空のとき）が未仕様、例外/ユーザーキャンセルのハンドリングが tasks にあるのに spec の Requirement に無いなど、Major 級の指摘が複数ある。

## Issues

### Issue 1: 複数画像（画像配列）の共有挙動が未定義
- **Severity**: Major
- **Location**: `specs/web-share/spec.md`, `proposal.md`
- **Line**: spec L19-25 / proposal L11-13
- **Description**: 対象コード `MainContent.svelte` の画像は `let images: ImageData[]`（配列）であり、`urlsToPost = images.map(img => img.croppedUrl ?? img.originalUrl)` のように複数画像を扱う。しかし proposal/spec は「画像がある場合はテキストと画像ファイルを共有する」と単数前提で記述され、複数画像選択時に全画像を `files` 配列で渡すのか、先頭1枚のみかが定義されていない。同じドメインの PPP-007 では画像ごとにコピーボタンを置く設計（個別1枚単位）になっており、本機能が「まとめて全画像」なのか方針が不明確。テスト可能性を欠く。
- **Suggestion**: Scenario「テキストと画像を共有する」に複数画像時の期待結果（例: 選択中の全画像が `files` 配列として渡される）を明記する。Requirement 本文でも `files: File[]` に全画像を含める旨を SHALL で定義する。

### Issue 2: 共有ボタンの活性条件（テキスト・画像とも空のとき）が未仕様
- **Severity**: Major
- **Location**: `specs/web-share/spec.md`, `tasks.md`
- **Line**: spec 全体 / tasks L3-7
- **Description**: 共有対象が何もない（テキスト空かつ画像なし）場合のボタン挙動が Requirement にも tasks にも無い。既存の投稿ボタンは `disabled={text.length <= 0 && images.length <= 0}` で制御されており、姉妹 proposal PPP-007 も「テキストが空のときは無効」を明示的に spec 化している。本 proposal は API 非対応時の無効化は定義するが、内容が空のときの扱いが欠落しており一貫性に欠ける。
- **Suggestion**: 「テキストも画像も無い場合は共有ボタンを無効化する (SHALL)」という Requirement と Scenario を追加し、tasks にも対応タスクを追加する。空内容で `navigator.share({})` を呼ばないことを保証する。

### Issue 3: 例外・ユーザーキャンセルのハンドリングが spec の Requirement に無い
- **Severity**: Major
- **Location**: `specs/web-share/spec.md`, `tasks.md`
- **Line**: tasks L23（4.2）
- **Description**: tasks.md 4.2 に「共有処理中の例外（ユーザーキャンセル含む）を適切にハンドリングする」とあるが、対応する Requirement/Scenario が spec に存在しない。Spec ↔ Tasks 網羅性が崩れており、また「適切にハンドリング」という語自体が曖昧（ユーザーキャンセル時は何もしない／エラー時は通知する、等の期待挙動が不定）。PPP-007 は「失敗を無言で握りつぶしてはならない (SHALL NOT)」と明確化している。
- **Suggestion**: spec に Requirement を追加し、(a) ユーザーがキャンセル（`AbortError`）した場合はエラー表示せず正常終了する、(b) それ以外の共有失敗時はユーザーに通知する、を Scenario として明記する。tasks.md 4.2 の「適切に」を具体的記述へ置換する。

### Issue 4: tasks.md「4.1 無効化または非表示」が spec と同じく二者択一のまま
- **Severity**: Minor
- **Location**: `proposal.md` L14 / `specs/web-share/spec.md` L40,L46 / `tasks.md` L22
- **Description**: 「無効化または非表示」という二択が proposal・spec・tasks すべてで未確定のまま残っている。仕様としては成立するが、実装者がどちらを選ぶかでユーザー体験（ボタンの存在が見えるか否か）が変わるため、テスト観点（Scenario L46 の THEN）が「無効化または非表示」と曖昧になっている。
- **Suggestion**: どちらか一方に決定するのが望ましい。決定しない場合は、なぜ実装裁量に委ねるのかの根拠を proposal に1行補足する。

### Issue 5: 成功時のユーザーフィードバックの有無が proposal/spec で未言及
- **Severity**: Minor
- **Location**: `proposal.md`, `specs/web-share/spec.md`
- **Description**: 姉妹 proposal PPP-007 は「コピー成功・失敗をユーザーにフィードバックする」を明記しているが、本 proposal は共有成功時のフィードバックに触れていない。Web Share では OS 共有シートが起動するため成功フィードバックは不要という判断はあり得るが、その判断が文書化されていないため意図が読み取れない。
- **Suggestion**: 「共有シート起動自体がフィードバックとなるため追加 UI フィードバックは設けない」等、非ゴールとして proposal に明記する（任意）。

### Issue 6: `croppedUrl ?? originalUrl` の表記がコード実体（`img.` プレフィックス・配列要素）と乖離
- **Severity**: Minor
- **Location**: `proposal.md` L12 / `tasks.md` L15
- **Description**: 実コードは `images.map(img => img.croppedUrl ?? img.originalUrl)` であり、`croppedUrl`/`originalUrl` は配列要素 `img` のプロパティ。proposal/tasks の表記は誤りではないが、Issue 1（複数画像）と相まって単数前提の誤読を招く。
- **Suggestion**: Issue 1 を解消する際、各画像について `img.croppedUrl ?? img.originalUrl` を Blob → File 変換し配列化する旨に表現を統一する。
