# Threads PR ゴースト投稿（準定期）実装計画

## Context

ユーザーは pppost から Threads へ投稿する際、自分の投稿に続けて PR（宣伝）内容を
**ゴースト投稿**（24時間で自動消滅する Threads 投稿）として自動付与したい。
頻度は「自分の投稿タイミングに便乗（不定期）」しつつ「前回 PR から48時間以上経過時のみ」
という下限ゲートを設けることで、**2日に1回程度の準定期**運用にする。

Threads API はゴースト投稿に対応している（コンテナ作成時に `is_ghost_post=true`）。
ただし**テキストのみ・リプライ不可・500文字まで**という制約がある。
そのため PR は「本投稿の直後の独立したゴースト投稿」として出す（リプライ不可のため）。

### 確定した仕様（ユーザー確認済み）

- **形式**: 本投稿の直後に独立したゴースト投稿として PR を出す
- **PR 文の管理**: 複数文を登録し、投稿のたびに**順番にローテーション**して1つ選ぶ
- **間隔判定**: localStorage に「前回 PR を出した時刻」を記録し、48時間（設定で変更可）以上経過時のみ自動付与
- **発火条件**: 投稿対象に **Threads が含まれているときのみ**（他 SNS には PR を出さない）

## 現状の把握

- Threads 投稿フロー: [MainContent.svelte](frontend/src/lib/MainContent.svelte) `post()` → [MainContent.ts](frontend/src/lib/MainContent.ts) `postToSns()`(170行) → `postToThreads()`(288行) → backend [threads_post.js](backend/netlify/functions/threads_post.js) でコンテナ作成→完了待機→publish の2段階フロー
- 状態は **localStorage のみ**（[func.ts](frontend/src/lib/func.ts) の `savePostSetting`/`loadPostSetting` 等）。バックエンド DB は無い
- テンプレート/定型文・スケジュール機能は**存在しない**ため新設する
- `is_ghost_post` はバックエンド未対応 → 追加が必要

## 実装方針

### 1. バックエンド: `is_ghost_post` 対応

**ファイル**: [backend/netlify/functions/threads_post.js](backend/netlify/functions/threads_post.js)

- `handler()`(94行)のリクエスト分解に `is_ghost_post` を追加（109行）
- ゴースト投稿はテキストのみのため、`is_ghost_post === true` の場合は画像を無視し
  TEXT コンテナ作成（127行のブロック）に `is_ghost_post: true` パラメータを付与する。
  既存の TEXT/IMAGE/CAROUSEL 分岐の前で「ghost なら強制 TEXT」とガードするのが安全。
- 完了待機(`waitForContainerReady`)・publish(`publishContainer`)は既存ロジックをそのまま流用。

### 2. フロントエンド: PR 設定と状態の保存

**ファイル**: [frontend/src/lib/func.ts](frontend/src/lib/func.ts)

既存の `savePostSetting`/`loadPostSetting` パターンに倣い、専用の型と関数を追加する
（PR 設定は SNS アカウント設定(`SettingData`)とは別系統なので独立させる）。

```typescript
export type PrGhostSetting = {
  enabled: boolean;
  intervalHours: number;   // 既定 48
  texts: string[];         // ローテーション対象の PR 文（各 ≤500文字）
};
export type PrGhostState = {
  lastPostedAt: number;    // 前回 PR を出した時刻（ms）。未投稿は 0
  rotationIndex: number;   // 次に使う texts のインデックス
};
```

- localStorage キー: `ppp_pr_ghost_setting` / `ppp_pr_ghost_state`
- `savePrGhostSetting` / `loadPrGhostSetting` / `savePrGhostState` / `loadPrGhostState` を追加
  （`loadMessage` 同様、未設定時は null を返す）

### 3. フロントエンド: 投稿後の PR 自動付与ロジック

**ファイル**: [frontend/src/lib/MainContent.ts](frontend/src/lib/MainContent.ts)

- `postToThreads()`(288行)に第4引数 `options?: { is_ghost_post?: boolean }` を追加し、
  リクエストボディに `is_ghost_post` を含める。
- `postToSns()`(170行)の本投稿ループ完了後（216行の `errors.length == 0` 判定の流れの中）に
  PR 付与ブロックを追加:
  1. `enableTypes` に `'threads'` が含まれ、かつ `errors` に `'Threads'` が**無い**（本投稿成功）
  2. `loadPrGhostSetting()` が `enabled === true` かつ `texts.length > 0`
  3. `Date.now() - state.lastPostedAt >= intervalHours * 3600_000`（未投稿は即 OK）
  → すべて満たす場合のみ:
     - `texts[rotationIndex % texts.length]` を選択
     - `postToThreads(prText, [], undefined, { is_ghost_post: true })` を実行
     - 成功時のみ `savePrGhostState({ lastPostedAt: Date.now(), rotationIndex: rotationIndex + 1 })`
- PR 投稿の失敗は**本投稿の成否に影響させない**（console にログのみ。`errors` には積まない）。
  本投稿成功後の付加機能のため、失敗してもユーザー投稿は成立しているとみなす。

### 4. フロントエンド: PR 設定 UI

**ファイル**: [frontend/src/lib/ThreadsConnection.svelte](frontend/src/lib/ThreadsConnection.svelte)

PR ゴースト投稿は Threads 専用機能のため、既存の Threads 設定（接続済み表示の下、69行付近）に
折りたたみセクションとして追加する。

- 有効/無効トグル（`enabled`）
- 間隔（時間）入力（`intervalHours`、既定 48）
- PR 文リスト（textarea を複数行 or 1行1文で管理。各文 500文字制限の注意表示）
- 変更時に `savePrGhostSetting()` で保存
- 既存の `loadPostSetting('threads')` が null（未接続）のときは PR 設定欄を非表示にする

## 検証方法

1. **ビルド**: `cd frontend && npm run build`、`cd backend && npm run build`（または lint）で型エラーが無いこと
2. **間隔ロジックの単体確認**: ブラウザ devtools で `ppp_pr_ghost_state` を操作し
   - `lastPostedAt` を 48時間以上前 → 投稿時に PR ゴーストが付く
   - `lastPostedAt` を直近 → 投稿しても PR が付かない
   を確認
3. **ローテーション確認**: texts に複数文を登録し、連続投稿（state をリセットしながら）で
   `rotationIndex` が進み、文が順番に切り替わること
4. **実投稿（要 Threads 接続）**: Threads を投稿対象にして本投稿 → 直後に PR がゴースト投稿として
   出ること、Threads アプリ上で吹き出し表示（24時間で消える）になっていること、
   PR にはリプライが付かない/テキストのみであることを確認
5. **発火条件**: Threads を投稿対象から外した状態では PR が出ないことを確認

## 留意点

- PR 文は 500文字・テキストのみ・リプライ不可というゴースト投稿制約を UI と送信前にチェックする
- 本投稿が複数 SNS でも、PR ゴーストは Threads のみに出る（仕様どおり）
- バックエンドの 10秒タイムアウト内に「本投稿 + PR 投稿」の2回が別リクエストとして走る
  （PR はフロントから2回目の `threads_post` 呼び出し）。各々が独立に完了待機するため問題なし
