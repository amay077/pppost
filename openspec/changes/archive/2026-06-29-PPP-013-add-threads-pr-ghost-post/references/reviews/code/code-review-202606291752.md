- **Implementation**: `backend/netlify/functions/threads_post.js:109`（body から is_ghost_post 抽出）、`:119-130`（is_ghost_post===true 分岐で media_type:'TEXT' + is_ghost_post:true、imageUrls 無視）、`:131-191`（else に従来の TEXT/IMAGE/CAROUSEL フローを温存）、`:193-203`（waitForContainerReady → publishContainer をゴースト/通常共通で実行し、失敗時 errorResponse を返す）。
- **Status**: OK
- **Note**: ゴースト分岐では replyParams を付与しない（リプライ不可制約に合致）。完了待ち・公開・失敗返却が共通経路で適用される点を確認。

##### Scenario 検証

#### REQ-002: PR ゴースト投稿設定の管理

#### REQ-003: PR ゴースト投稿の自動付与
- **Spec**: 本投稿成功かつ「Threads が投稿対象 × Threads 本投稿成功(errors に Threads を含まない) × enabled × texts>=1 × 間隔経過」をすべて満たすときのみ PR を 1 つゴースト投稿。登録順ローテーション。成功時のみ lastPostedAt/rotationIndex 更新、失敗時は未更新。PR 失敗は errors/成功通知に含めない。
- **Implementation**: `MainContent.ts:219`（発火ガード `enableTypes.includes('threads') && !errors.includes('Threads')`、errors.length==0 とは独立）、`:223`（画像削除等の後処理は従来どおり errors.length==0 で判定し混同なし）。`tryPostPrGhost`（`:254-281`）: `:257` enabled!==true / texts.length<=0 で return、`:262-266` 間隔判定 `Date.now()-lastPostedAt < intervalHours*3600_000`（未投稿 0 は常に経過）、`:270` `texts[rotationIndex % texts.length]` でローテーション、`:271` postToThreads(prText, [], undefined, { is_ghost_post:true })、`:273-277` 成功時のみ savePrGhostState、失敗時は console のみで state 未更新、全体 try/catch で例外も errors に伝播しない。
- **Status**: OK

##### Scenario 検証
- 間隔経過後に PR が付与される: OK（間隔判定通過 → postToThreads ghost → savePrGhostState で lastPostedAt/rotationIndex 更新）
- 間隔内では PR が付与されない: OK（`:266` で return）
- Threads が投稿対象でなければ付与しない: OK（`:219` の enableTypes.includes('threads') ガード）
- Threads 本投稿が失敗したら付与しない: OK（`:219` の !errors.includes('Threads') ガード。tryPostPrGhost を呼ばないため state 更新もなし）
- PR 文がローテーションする: OK（rotationIndex を成功ごとに +1、% texts.length で循環）
- PR 投稿失敗は本投稿に影響しない: OK（tryPostPrGhost は errors を一切操作せず、戻り値 void。失敗時 state 未更新で次回再試行）

### Minor Issues

#### Issue 1: 間隔判定が intervalHours<0 を許容（軽微）
- **Severity**: Minor
- **Location**: `frontend/src/lib/ThreadsConnection.svelte:99-100`
- **Description**: intervalHours 入力は `min="0"` だが負値や 0 の妥当性は明示制約されていない。0 の場合は毎回 PR が付与される（spec は既定 48 を要件化しているが下限は未規定）。spec で明記されていないため判定対象外に近いが、想定外の運用を招きうる。
- **Spec Reference**: REQ-002（付与間隔）
- **Suggestion**: 必須ではない。運用上 0 や負値を防ぎたい場合のみ下限バリデーションを検討。

#### Issue 2: rotationIndex が際限なく増加（軽微）
- **Severity**: Minor
- **Location**: `frontend/src/lib/MainContent.ts:274`
- **Description**: rotationIndex は `% texts.length` で参照されるため動作上は正しいが、保存値自体は増加し続ける。実用上の問題はない。
- **Spec Reference**: REQ-003（ローテーション）
- **Suggestion**: 必須ではない。気になる場合のみ保存時に剰余を取る。
