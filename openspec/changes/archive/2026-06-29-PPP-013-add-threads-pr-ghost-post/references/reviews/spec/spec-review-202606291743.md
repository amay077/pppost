## Minor Issues

### Issue 1: 「本投稿成功」の判定基準が既存コードの `errors.length == 0` と異なる点の明示
- **Severity**: Minor
- **Location**: `design.md` D5 / `specs/threads-posting/spec.md` L60 / `tasks.md` 3.2
- **Description**: spec は PR 発火条件を「エラー一覧に `Threads` を含まない」（= Threads 単体の成功）と定義している。一方、既存 `MainContent.ts` の本投稿成功後処理（画像削除・設定保存）は L216 の `errors.length == 0`（= 全 SNS 成功）を判定に使っている。両者は意図的に異なる基準であり仕様上は正しい（他 SNS が失敗しても Threads が成功していれば PR を出す）が、実装者が既存ブロックの条件を流用すると誤判定する恐れがある。spec/tasks のどちらも「全 SNS 成功ではなく Threads 単体の成功で判定する」点は読み取れるため矛盾ではないが、design.md に「既存の `errors.length == 0` ブロックとは別条件である」旨を一文補足すると実装ミスを防げる。
- **Suggestion**: design.md D5 に「PR 判定は既存の `errors.length == 0`（全 SNS 成功）ブロックとは独立し、`errors` に `Threads` を含まないことのみで判定する」と明記する。

### Issue 2: PR ゴースト投稿のコンテナ完了待機タイムアウト失敗時の挙動が spec に未記載
- **Severity**: Minor
- **Location**: `specs/threads-posting/spec.md` L15, L68
- **Description**: ゴースト投稿は「公開前のコンテナ処理完了待ち」要件が適用され、公開失敗時は呼び出し元へ返す（L15）とある。一方、PR ゴースト投稿要件（L68）では「PR の失敗は本投稿の成否へ影響させず、`errors` に含めない」としている。両者を突き合わせると「ゴースト投稿バックエンドは失敗を返すが、フロントの PR 付与ロジックがそれを握り潰して `console` ログのみとする」という処理経路になり、整合は取れている。ただし、コンテナ完了待機がタイムアウトした場合に `lastPostedAt` を更新しない（次回再試行される）点は D3/D4 と tasks 3.5 で担保されているものの、spec 本文には「失敗時は状態未更新」のみ記載で、タイムアウト失敗も同じ扱いになるかが明示的でない。Scenario「PR 投稿失敗は本投稿に影響しない」(L102-107) でカバーされているため致命的ではない。
- **Suggestion**: 任意。L68 付近に「公開失敗・完了待機タイムアウトを含む PR 投稿のあらゆる失敗で状態を更新しない」と一文加えると網羅性が増す。
