# 投稿トークンのサーバー側保管（BFF 化）計画 — 確定版

> 旧タスク（Threads PR ゴースト投稿=PPP-013）は完了・アーカイブ済み。暫定案（案3=暗号化のみ）は破棄。
> 本ファイルは「投稿用トークンをサーバー側のみで保持する」根本設計の確定版。

## Context（なぜ）

各 SNS のアクセストークン（Threads=60日長命、Bluesky=session JWT 一式、Mastodon=access_token）が現状
**クライアントの localStorage に平文保存され、操作ごとに平文でサーバーへ往復**している。XSS で全 SNS
トークンが一括流出し、pppost 外での恒久的アカウント乗っ取りに直結する。

根本解は **BFF**: トークンをサーバー側の保管庫（Cloudflare D1）に暗号化保存し、**クライアントには一切返さない**。
クライアントは pppost が発行する**不透明なセッション ID だけ**を保持し、Bearer で認可する。

## 確定した方針

- **セッション伝送: 案A Bearer**（`Authorization: Bearer <session_id>`）。現構成（GitHub Pages + Netlify、クロスサイト）を維持。
- **利用形態: 個人単一ユーザー**。ログイン UI なし。端末ごとの匿名セッション（サーバー発行の高エントロピー ID）で保管庫と紐付ける。
- **保管先: Cloudflare D1**（Netlify Functions から D1 HTTP API 経由。CF API トークン/DB ID は Netlify 環境変数）。
- SNS トークンは server-only 鍵（既存 `PPPOST_DATA_SECRET`、レコードごとランダム IV）で **AES 暗号化して D1 に保存**。

## 現構成の制約（確認済み）

- フロント=GitHub Pages / API=Netlify Functions の**クロスサイト**（CORS `*`）→ Cookie は使わず Bearer とする。
- サーバー DB は未導入（R2 は画像用のみ）→ D1 を新設。D1 に**ネイティブ TTL はない**（`expires_at`/`last_*_at` カラム +
  時刻比較でエミュレート。掃除が要れば Cloudflare Cron Triggers）。
- ログイン/セッションの足場なし → 匿名セッションを新設。crypto と鍵材料は健在。

## ターゲット・アーキテクチャ

### D1 スキーマ（想定）
- `sns_credentials(session_id TEXT, sns_type TEXT, enc_token TEXT, meta TEXT, updated_at INT, PRIMARY KEY(session_id, sns_type))`
- `pr_ghost_state(session_id TEXT PRIMARY KEY, last_posted_at INT, rotation_index INT)`（PPP-013 の localStorage 状態を移管）
- PR 設定（有効/間隔/文リスト）も D1 に寄せるかは設計で判断（同一 session_id で統合可）。

### セッション確立フロー（匿名・サーバー発行）
1. 最初の SNS 接続時、client にセッション ID が無ければ、backend がトークン保存時に**高エントロピーな session_id を生成**し
   D1 に行を作成、**session_id だけ**を返す。client は `ppp_session_id` として保存。
2. 以降の全 API 呼び出しは `Authorization: Bearer <session_id>`。backend は D1 で行を引き、必要な SNS トークンを復号して使用。
3. 追加 SNS 接続は同じ session_id の行に該当 SNS トークンを追記。

### トークンをクライアントへ返さない
- 発行側 [threads_token.js](backend/netlify/functions/threads_token.js) / [mastodon_token.js](backend/netlify/functions/mastodon_token.js) / [bluesky_login.js](backend/netlify/functions/bluesky_login.js):
  トークンを返さず **D1 に暗号化保存**。返すのは session_id と表示用メタ（handle 等）のみ。
- 消費側 threads_post/threads_posts/threads_refresh/mastodon_post/mastodon_posts/bluesky_post/bluesky_posts:
  client からの token 受領を廃止し、**session_id から D1 のトークンを引く**。
- トークンリフレッシュ（Threads 24h、Bluesky `refreshSession`）を**完全サーバー側へ移管**。

### ゴースト投稿間隔のサーバー判定（PPP-013 移管）
- `pr_ghost_state` を D1 に持ち、`now - last_posted_at >= interval` を**サーバーで判定**して発火（改ざん不可・多端末一貫）。

## 影響範囲（コード）

- バックエンド [backend/netlify/functions](backend/netlify/functions):
  - D1 アクセス + AES 暗号化の共通ユーティリティ新設、セッション発行/検証の共通処理
  - 発行側 3 関数・消費側 7 関数を上記のとおり改修
- フロントエンド:
  - [func.ts](frontend/src/lib/func.ts): `ppp_setting_*` から**トークンを除去**（接続有無＋表示メタのみ）、`ppp_session_id` を追加
  - [MainContent.ts](frontend/src/lib/MainContent.ts): 各 `postToXxx` の token 送信を廃止し、Bearer ヘッダを付与
  - 各 `*Connection.svelte`: OAuth コールバック後にトークンを受け取らない流れへ（Threads redirect 着地点の見直し含む）

## 移行

既存 localStorage の平文トークンは破棄し、**各 SNS を再接続**（D1 に入り直す）。後方互換は持たない。

## 検証方法

- 接続・投稿後、**localStorage とネットワークに SNS トークンが一切現れない**ことを DevTools で確認（現れるのは session_id のみ）
- 3 SNS の投稿・自投稿取得・Threads リフレッシュが D1 保管トークンで成立
- ゴースト投稿間隔がサーバー判定で効く（localStorage 改変で回避できない）
- `cd frontend && npm run build` 通過、D1 HTTP API 疎通
