### 詳細

#### CC-1: サーバー発行の匿名セッション
- **Implementation**: `lib/session.js`（`crypto.randomBytes(32).toString('hex')`、`^Bearer\s+(.+)$` 抽出）。発行系 3 関数が `extractSessionId(event) ?? generateSessionId()` で「既存なら再利用・無ければ新規発行」。フロントは各接続で `saveSessionId(resJson.session_id)`、`func.ts` の `ppp_session_id` に保存。
- **Status**: OK
- **Scenario**「初回発行」「追加接続で再利用（UPSERT ON CONFLICT(session_id, sns_type)）」「無効セッション拒否」いずれも成立。無効セッションは `pr_ghost_setting` では `sessionExists()`→401、消費系では `getToken`→null→4xx で「認可されず・トークン不使用」に帰着。

#### CC-2: SNS トークンのサーバー暗号化保管
- **Implementation**: `lib/crypto.js` は `crypto.randomBytes(16)` の**ランダム IV** を毎回生成し `${ivHex}:${cipherHex}` で保存。`PPPOST_DATA_IV`（固定 IV）不使用を全 backend で確認。鍵は `PPPOST_DATA_SECRET` の先頭 32B（サーバー環境変数のみ）。発行系レスポンス body は `threads_token`={session_id,user_id} / `mastodon_token`={session_id,server} / `bluesky_login`={session_id,handle,did}。meta にトークン成分なし。
- **Status**: OK

#### CC-2（露出しない）: フロント localStorage
- **Implementation**: `func.ts` の `SettingData*` 型はトークンフィールドを持たない（threads=user_id, mastodon=server, bluesky=handle/did のみ）。`savePostSetting` はこの型のみを保存。旧 `access_token`/session 保存経路は grep で不在。
- **Status**: OK

#### CC-3: 保管トークンによる操作の実行
- **Implementation**: `threads_post/threads_posts/threads_refresh/mastodon_post/mastodon_posts/bluesky_post/bluesky_posts` すべてが `extractSessionId`→null で 401、`getToken(sessionId, sns)`→null で 4xx（threads/mastodon/bluesky=400、refresh=200 no-op）。トークンは常に保管庫から復号取得。クライアント body の token/host/sessionData 受領経路は撤去済み（`MainContent.ts` は Bearer のみ送信）。
- **Status**: OK

#### CC-4: 保管トークンの削除（切断）
- **Implementation**: `sns_disconnect.js` は Bearer 必須（null→401）、`SNS_TYPES` 検証、`deleteToken(sessionId, sns_type)` が `DELETE ... WHERE session_id=? AND sns_type=?` で当該行のみ削除。他 SNS 行は保持。3 SNS 共通適用。フロントは各 `*Connection.svelte` の `onDisconnect` が呼び出し後 `deletePostSetting` で UI 未接続化。
- **Status**: OK

#### TP-1: Threads アカウント接続
- **Implementation**: `ThreadsConnection.svelte` の authorize は `scope=threads_basic,threads_content_publish,threads_manage_replies`（3 スコープ）。`MainContent.svelte` onMount が `threads_callback` を捕捉し `threads_token?code=` を叩き、`session_id`/`user_id` のみ保存。`threads_token.js` は短命→長命交換→`saveToken('threads', {...}, {user_id})`。client_secret はサーバーのみ。OOB 不使用（redirect_uri 遷移）。
- **Status**: OK

#### TP-2: 長命トークン自動リフレッシュ
- **Implementation**: `threads_refresh.js` が `Date.now()-stored.updatedAt < 24h` で no-op、経過時のみ `th_refresh_token` 実行し `saveToken` で更新（updated_at=Date.now()）。失敗時は `{refreshed:false}` を返しトークン非削除・接続維持。新トークンは返却しない。判定・実行はサーバー側。フロント onMount は接続時に `threads_refresh` を叩くだけ（obtained_at 判定は撤去）。
- **Status**: OK

#### TP-3: PR ゴースト投稿設定の管理
- **Implementation**: `pr_ghost_setting.js`（GET/PUT）が Bearer+`sessionExists`→401、`pr-ghost.js` の `savePrGhostSetting` が D1 UPSERT（実行状態は初回のみ初期化・更新時保持）。既定間隔 48。`ThreadsConnection.svelte` は接続時（postSettings!=null）のみ UI 表示、500 字超過を warning 表示。localStorage 保存なし。
- **Status**: OK

#### TP-4: PR ゴースト投稿の自動付与
- **Implementation**: `threads_post.js` の `tryPostPrGhost` は本投稿 `result.ok` 後にのみ呼ばれる。判定: `enabled===true && texts.length>0` かつ `lastPostedAt==null || (now-lastPostedAt >= intervalHours*3600_000)`（未投稿=経過扱い）。選択: `rotationIndex % texts.length`（登録順ローテーション）。成功時のみ `updatePrGhostExecState(now, rotationIndex+1)`。失敗/例外は console のみで本投稿レスポンス（`{}` 200）に非影響。`is_ghost_post` はサーバー側ゴーストパスでのみ付与、クライアント指定は撤去（frontend に is_ghost_post 参照なし、`MainContent.ts` はクライアント発火を廃止）。
- **Status**: OK

## Issues

### Issue 1: 消費系の無効セッションが 401 でなく 400 に集約される
- **Severity**: Minor
- **Location**: `backend/netlify/functions/threads_post.js:244-247`, `threads_posts.js:27-30`, `mastodon_post.js:32-39`, `mastodon_posts.js:31-38`, `bluesky_post.js:73-80`, `bluesky_posts.js:34-41`
- **Description**: 消費系は「存在しない session ID（未発行/削除済み）」と「有効 session だが当該 SNS 未保管」をいずれも `getToken`→null→400（refresh は 200 no-op）として扱う。`pr_ghost_setting.js` は `sessionExists()` で無効セッションを 401 に分けており、扱いが不統一。ただし spec の要件（「認可されず、トークンは使用されない」「未保管は失敗として扱う」）はどちらも満たしており、機能的な齟齬はない。
- **Spec Reference**: credential-custody「無効なセッションは認可されない」/「未保管の SNS への操作は失敗する」
- **Suggestion**: 要件充足済みのため必須ではない。ステータスの一貫性を重視するなら消費系でも無効セッションを 401 に分けると明確。

### Issue 2: 「1 行でもあればセッション有効」定義と PR 設定エンドポイントの整合
- **Severity**: Minor
- **Location**: `backend/netlify/functions/pr_ghost_setting.js:25-27`
- **Description**: `sessionExists()` は sns_credentials に 1 行でもあれば true を返すため、Bluesky のみ接続で Threads 未接続でも PR 設定 API が 200 を返しうる。spec の「未接続時は設定 UI を表示しない」は UI 要件で `ThreadsConnection.svelte`（postSettings!=null 判定）が担保しているため spec 違反ではないが、backend は Threads 保管有無を検査していない。design.md D3 の「保管レコード存在＝セッション有効」定義とは整合。
- **Spec Reference**: threads-posting「未接続時は設定 UI を表示しない」（UI 要件）
- **Suggestion**: 厳格化するなら backend で `getToken(sessionId,'threads')` の存在も条件化。UI ゲートで足りるため任意。

### Issue 3: 空文字の PR 文でもゴースト投稿を試行しうる
- **Severity**: Minor
- **Location**: `backend/netlify/functions/threads_post.js:199,210-213`
- **Description**: `texts.length > 0` のみを条件とするため、`texts=['']`（空文字）でも `prText=''` でゴースト投稿を試行する。spec は「PR 文が 1 つ以上」としており空文字を除外する明示要件はない。
- **Spec Reference**: threads-posting「PR 文が 1 つ以上」
- **Suggestion**: 必要なら空/空白のみ要素を除外して判定。
