## Summary

BFF 化による投稿トークンのサーバー側保管という意図は proposal / design / spec / tasks / plan で一貫しており、`openspec validate --strict` もパスする。MODIFIED 見出し 4 件は既存 spec と完全一致し、実装コード（token 受領・session 返却）とも噛み合う。ただし MODIFIED の PR ゴースト投稿要件で「間隔既定 48 時間」「未投稿時は経過とみなす」という既存の正規化された内容が欠落しており、内容の欠落なき引き継ぎに反する（Major）。

## Major Issues

### Issue 1: MODIFIED で PR ゴースト投稿要件の正規化された内容が欠落している
- **Severity**: Major
- **Location**: `specs/threads-posting/spec.md`（MODIFIED: PR ゴースト投稿設定の管理 / PR ゴースト投稿の自動付与）
- **Line**: 60, 76
- **Description**: MODIFIED は要件全体を置換するため既存内容を欠落なく引き継ぐ必要があるが、以下 2 点が既存正式 spec（`openspec/specs/threads-posting/spec.md` L280, L304）から脱落している。
  1. 付与間隔の既定値「（時間、既定 48）」— 具体値が消え、間隔の既定が未定義になった。
  2. 「（未投稿時は経過とみなす）」— 新規セッションで `last_posted_at` が未設定（NULL）のときに初回 PR を発火するか否かが未定義になった。D1 移管後は新規レコードの初期状態が必ず発生するため、この初回挙動の未定義は実装分岐で解釈が割れる functional gap。
- **Suggestion**: MODIFIED 要件本文に「付与間隔（時間、既定 48）」と「前回 PR を出した時刻から設定間隔以上経過している（`last_posted_at` 未設定時は経過済みとみなす）」を明記し、初回セッションの発火挙動を検証する Scenario を 1 つ追加する。

## Minor Issues

### Issue 2: MODIFIED で複数の既存 Scenario が失われ、負の挙動のテスト可能性が低下
- **Severity**: Minor
- **Location**: `specs/threads-posting/spec.md`（MODIFIED: Threads 長命トークンの自動リフレッシュ / PR ゴースト投稿の自動付与）
- **Description**: 正規の SHALL/SHALL NOT 文自体は本文に保持されているが、既存 spec にあった以下の Scenario が MODIFIED から消えた: リフレッシュ「未接続時は何もしない」、自動付与「Threads が投稿対象でなければ付与しない」「Threads 本投稿が失敗したら付与しない」「PR 文がローテーションする」「PR 投稿失敗は本投稿に影響しない」。本文の規範は残るためフォーマット上は最低 1 Scenario を満たすが、安全側（付与しない）分岐の検証カバレッジが低下する。
- **Suggestion**: 少なくとも「Threads 本投稿失敗時に付与しない」「PR 失敗が本投稿に影響しない」の 2 つは、サーバー判定文脈へ書き換えたうえで Scenario として残す。

### Issue 3: credential-custody に切断（トークン削除）要件が定義されていない
- **Severity**: Minor
- **Location**: `specs/credential-custody/spec.md`
- **Description**: threads-posting MODIFIED の切断 Scenario（L25）は「サーバー保管庫から当該セッションの Threads トークンが削除される」を前提とするが、credential-custody の ADDED 3 要件（セッション発行 / 暗号化保管 / 保管トークンでの操作）には削除・切断の要件がない。tasks 2.4 に「削除の共通関数」はあるが、capability レベルの要件が欠落。Mastodon / Bluesky の切断時トークン削除もどの spec でも規定されていない。
- **Suggestion**: credential-custody に「保管トークンの削除（切断）」要件を追加し、SHALL で「切断要求時に該当セッション×SNS の保管トークンを削除する」を定義する。

### Issue 4: セッションの有効期限・失効条件が未定義
- **Severity**: Minor
- **Location**: `specs/credential-custody/spec.md`（サーバー発行の匿名セッション）
- **Description**: Scenario「無効なセッションは認可されない」で「失効した session_id」に言及するが、セッションがいつ・どの条件で失効するか（TTL・無期限か）が要件本文に定義されていない。design.md D2 は「D1 にネイティブ TTL なし、`expires_at`/`last_*_at` でエミュレート、必要なら Cron パージ」とするが、spec 側に失効の規範がない。
- **Suggestion**: セッション失効を採用するなら失効条件を要件化する。無期限運用なら Scenario の「失効した」文言を「存在しない（削除済み含む）」に整えて曖昧さを消す。

### Issue 5: OAuth スコープ文字列の変更が What Changes に明記されていない
- **Severity**: Minor
- **Location**: `specs/threads-posting/spec.md`（MODIFIED: Threads アカウント接続）/ `proposal.md`
- **Description**: MODIFIED 接続要件は scope を `threads_basic,threads_content_publish,threads_manage_replies` に変更している。既存の接続要件本文は 2 スコープ表記（L8）で、未修正のリプライ要件（L220）は 3 スコープを要求しており、本 MODIFIED は latent な不整合を是正する望ましい変更。ただし proposal の What Changes はトークン保管のみを謳っており、スコープ是正が変更意図として明記されていない。
- **Suggestion**: What Changes に「接続要件の scope 表記をリプライ要件と整合（3 スコープ）に是正」を 1 行追記する。挙動意図としては問題なし。
