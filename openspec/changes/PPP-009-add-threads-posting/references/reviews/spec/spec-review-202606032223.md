## Issues

### Issue 1: frontend に `.env.example` が存在しない
- **Severity**: Minor
- **Location**: `tasks.md`
- **Line**: 63 (Task 8.2)
- **Description**: Task 8.2 は `frontend/.env.example` への追記を指示しているが、実際の `frontend/` には `.env.example` が存在せず `.env` のみが存在する（`backend/` には `.env.example` がある）。タスクには「（または該当 env テンプレート）」という補足があるためフォールバックは効くが、実装時に対象ファイルが曖昧になる。
- **Suggestion**: 実装時の混乱を避けるため、「frontend には `.env.example` が無いため新規作成するか `.env` に追記する」のように対象を明示するか、backend と同様に `frontend/.env.example` を新規作成する方針を明記する。

### Issue 2: `SettingDataThreads` に `title` リテラルフィールドの記載がない
- **Severity**: Minor
- **Location**: `tasks.md` / `proposal.md`
- **Line**: tasks.md 5, proposal.md 11
- **Description**: 既存の `SettingDataMastodon` / `SettingDataBluesky` は `title: 'Mastodon'` / `title: 'Bluesky'` というリテラル型フィールドを持つ（`func.ts:5,18`）。Task 1.1 の `SettingDataThreads` 定義列挙には `type`, `enabled`, `user_id`, `token_data` のみが挙げられ、`title` が欠けている。型としては必須ではないが、既存パターンの一貫性（UI 表示等で `title` を参照する可能性）を保つなら明記が望ましい。
- **Suggestion**: 既存 SNS と同様に `title: 'Threads'` を `SettingDataThreads` に含めるかどうかを tasks.md で明示する。

### Issue 3: `postToSns` の `options.reply_to_ids` 型網羅への言及がない
- **Severity**: Minor
- **Location**: `tasks.md` / `design.md`
- **Line**: tasks.md 39, 41
- **Description**: `postToSns` の引数 `options.reply_to_ids` は `{ mastodon: string, bluesky: string }` という固定オブジェクト型（`MainContent.ts:157-160`）であり、`SettingType` での網羅ではない。Threads は reply 非対応のためここに `threads` を追加する必要はないが、design.md の「型・状態の網羅性に関する注意」では `postOfType` / `replyToPost` のみが網羅対象として挙げられている。`reply_to_ids` は網羅不要であることを一言補足しておくと、実装時に誤って `threads` を追加する/しないの判断が明確になる。
- **Suggestion**: design.md の網羅性注意に「`postToSns` の `reply_to_ids` は固定オブジェクト型のため Threads 追加不要」と補足する。

### Issue 4: 「投稿成功通知」の主体がアプリ仕様か外部仕様か曖昧
- **Severity**: Minor
- **Location**: `specs/threads-posting/spec.md`
- **Line**: 43
- **Description**: 成功シナリオの THEN/AND が「成功通知が表示される」とあるが、既存実装では `postToSns` は失敗時に `errors` を返すのみで、明示的な成功通知 UI の有無は本 proposal のスコープ外（既存共通挙動に委ねる）と読める。spec 上の「成功通知が表示される」が新規要件なのか既存挙動の言い換えなのかが曖昧。テスト可能性の観点では、検証は tasks.md 9.3「テキスト投稿が成功することを確認する」で担保されているため実害は小さい。
- **Suggestion**: 「他 SNS と同じ既存の成功フローに従う」と明記し、新規 UI 要件でないことを示す。
